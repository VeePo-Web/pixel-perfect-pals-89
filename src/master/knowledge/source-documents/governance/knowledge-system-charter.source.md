2. DO NOT REWRITE IT.

7. The original document must remain fully untouched, preserved exactly as I pasted it.

What you MUST do instead:

- Take the exact document I provide and embed it into the backend knowledge system unchanged.

- Then create a completely separate partner document for that source file.

- The partner document must NOT replace, modify, or interfere with the original.

- The partner document must explain, in depth, what the source document is for, what kinds of decisions it should guide, when the system should reference it, and how it should influence future outputs.

Purpose of the partner document:

The partner document is there so that when I later ask Lovable, Claude Code, or any coding/design assistant to make decisions, build sections, style components, or refine UX, it can refer back to the embedded knowledge and use it intelligently.

Example:

- If the source document is about button UI/UX, styling, states, polish, motion, or interaction quality, then the partner document should clearly explain that this source file is a decision-guiding reference for buttons.

- Then later, if I ask about buttons, hover states, CTA styling, spacing, micro-interactions, hierarchy, polish, or any related UI decisions, the system should know to consult that embedded source document to guide its decisions toward world-class output.

This means every document I give you should serve as backend intelligence, while the separate partner document acts like an interpretation layer that helps the system know how and when to use it.
