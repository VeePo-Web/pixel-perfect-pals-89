## 2. What this document is

The backend-aware evolution of CMB strategy. Where 1.0 and 1.2 articulate CMB as a single brand, 1.3 articulates how CMB operates as a **system of sister trade sites**: shared backend, shared brand DNA, shared SEO moat, inherited design language — while each remix stays bespoke at the front end. Also frames legacy / generational positioning at an operational (not just copy) level.

## 8. Rules — non-negotiables this doc enforces

1. **One source of truth per concept.** If a concept is shared (brand voice, NAP, license, warranty, sister-network list), it lives in `src/master/`. If it's per-trade, it lives in `src/config/trade.config.ts`. No duplication. (source: system architecture)
2. **Inheritance is explicit.** Every remix records which master version it forked from (see `src/master/VERSION.ts`). Implicit drift is forbidden.
3. **Bespoke front, shared back.** Visual + copy front-end is bespoke per remix; backend (DB schema, edge functions, email infra, schema.org structure) is shared. Never invert.
4. **Sister-site links never look templated.** When a footer references the network, the language is bespoke per remix and never lists every sister verbatim. (source: footer / cross-site trust)
5. **NAP triplet (name, address, phone) propagates from master with override capability.** A remix may override; if it does, the override is recorded in `trade.config.ts`, never silently. (source: trust propagation)
6. **License + insurance numbers are real.** Master holds the canonical numbers; remixes inherit unless legally distinct entities. Placeholder numbers fail the launch gate. (cross-link to `LEGAL_TRUST_GUIDE.md`)
7. **Backlink network is curated, not exhaustive.** A given remix links to *contextually adjacent* sister sites only — not the entire roster. (source: inter-site SEO)
8. **50-year horizon test.** For backend decisions (schema names, taxonomy, infra), ask: "would this still make sense if CMB ran 30 trades in 2050?" If no, redesign. (source: "what survives")
9. **Taxonomy stability.** Trade slugs in `src/master/trades.ts` are versioned and append-only. Renaming a slug requires a migration plan. (source: system architecture)
10. **No remix inherits another remix.** Remixes inherit from master only. Sister-to-sister copying is forbidden — that path leads to `gr-zero-sister-fingerprints` violations.
11. **Legacy framing operates at the system level.** "Generations after us" applies to *the network*, not just one site. A founder section on a remix can reference the parent legacy without restating it. (source: legacy section)
12. **Cite 1.3 in the deep plan whenever multi-trade implications exist.** Even single-trade decisions that touch shared infra trigger this rule.

## 9. Anti-patterns

- A "Drywall by Cochrane" footer listing all 27 future trades. Violates Rules 4 + 7.
- License number `XXX-PLACEHOLDER` shipping to production. Violates Rule 6.
- Trade slug renamed from `roof-replacement` to `roofing-replacement` mid-flight with no migration. Violates Rule 9.
- Drywall remix copies the plumbing remix's hero verbatim with `s/plumbing/drywall/`. Violates Rule 10 + `gr-zero-sister-fingerprints`.
- A new "AI house designer" feature added to one remix's bespoke layer that should have been a master capability. Violates Rule 1 + Rule 3 (inverted).
- A remix's brand bible drifting from master without recording the fork version. Violates Rule 2.

## 10. Worked example

**Trigger.** Request: "Launch a new Electrical sister site."

**Step 1 — recognize.** "New sister site" → this doc.

**Step 2 — read source.** Parse PDF. Read system architecture + inherit/bespoke split + inter-site SEO sections.

**Step 3 — apply rules.** Rule 1: confirm electrical-specific config goes in `trade.config.ts`, not master. Rule 2: stamp `VERSION.ts` fork. Rule 3: bespoke front (electrical hero, palette accent, copy), shared back (booking infra). Rule 5: NAP inherits unless the entity is separate. Rule 6: real license + insurance for the electrical entity. Rule 7: footer links to *contextually adjacent* sisters (e.g. drywall, lighting), not all 27. Rule 9: slug `electrical` added append-only. Rule 11: legacy framing references parent network without restating it.

**Step 4 — decision.** Scaffold the remix per the master playbook (`REMIX_PLAYBOOK.md`), gate launch on real license number + curated backlink list + recorded fork version.

**Step 5 — verify.** Plan §12 lists `gr-zero-sister-fingerprints`, `gr-real-business-signals`, `gr-master-logo-slot-map`.
