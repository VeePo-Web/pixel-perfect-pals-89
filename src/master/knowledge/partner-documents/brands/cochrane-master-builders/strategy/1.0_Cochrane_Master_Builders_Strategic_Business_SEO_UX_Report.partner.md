## 2. What this document is

The first full strategic articulation of CMB: who it serves, what business it is in, the North Star, and the highest-level SEO + UX implications. Treat as **constitutional**: broad, foundational, deliberately abstract. It is the on-mission/off-mission test for every page, feature, and remix.

## 6. Routing precedence

- Loses to `strategy/1.2` on positioning where they disagree (1.2 is the iterated revision).
- Loses to `strategy/1.3` on multi-trade / backend / sister-site questions.
- Wins over personas, SEO research, and UX design when the question is "is this on-mission at all?"

**Conflict example.** A request says "let's add a finance-calculator widget to the home hero." `ux-design/1.3.1` might allow the layout; `seo-research/1.1` might find search demand. This doc decides whether financing belongs in CMB's North Star — if not, the widget fails the on-mission test regardless of the other two.

## 8. Rules — non-negotiables this doc enforces

1. **On-mission test first.** Before any new page, section, or feature ships, state in one sentence how it serves the North Star defined in this source. If the sentence is vague, the artifact is off-mission. (source: "what we promise")
2. **Never invent a North Star.** If the source's North Star is unclear for a given remix, surface the gap and ask — do not paraphrase a substitute. (source: executive summary)
3. **One promise per hero.** Top-level hero pitches make exactly one promise; supporting promises live below the fold. (source: opening positioning)
4. **Trade remixes inherit the North Star, never override it.** A drywall site can adapt the *expression* of the promise but cannot replace the promise itself. (source: "who we serve")
5. **No off-mission features.** Calculators, configurators, marketplaces, blogs, and tools must each pass the on-mission test before scoping. (source: "what we promise")
6. **No quoting source prose into the front end.** This is a backend strategy document; its *implications* shape pages, its *sentences* never appear in pages. (constraint of this partner-doc layer)
7. **When 1.0 conflicts with 1.2 or 1.3, defer.** This doc is constitutional but superseded by later iterations on the same point. State the conflict in the plan and follow the newer doc. (routing precedence)
8. **Strategic claims need backing.** If this doc is cited as the basis for a UX or copy decision, name the section consulted in the plan's Brand-truth refs (per `PLAN_FIRST_DISCIPLINE.md` §2).

## 9. Anti-patterns

- A hero that lists three different value propositions of equal weight ("custom homes, renovations, AND financing") — violates Rule 3.
- A new "AI design tool" page added because it's trendy, with no mission justification — violates Rule 1.
- A trade remix that swaps "build for those who come after us" for "fastest drywall in town" — violates Rule 4.
- A founder section that quotes paragraphs from the source PDF verbatim into the page — violates Rule 6.

## 10. Worked example

**Trigger.** Request: "Add a 'Financing Options' page to the main nav."

**Step 1 — recognize.** Triggers "should this page exist" and "is this on-mission" → this doc.

**Step 2 — read source.** `document--parse_document` on the PDF. Read "what we promise" + "who we serve" together.

**Step 3 — apply rules.** Rule 1 demands a one-sentence on-mission justification. If the source's North Star is "build foundations for those who come after us," financing is *adjacent* to legacy (multi-generational affordability) but not core. Rule 5 demands financing pass the test before scoping.

**Step 4 — decision.** Do not add financing to the main nav at this stage. Instead, write a one-paragraph "is financing on-mission?" memo for the operator, propose a footer link as a soft-launch, and gate full nav promotion on operator confirmation. Cite Rules 1 + 5 in the plan.

**Step 5 — verify.** The plan's Section 12 (Guard Rail Compliance) lists `gr-plan-first-deep-items` because the deferred decision requires the deep-plan template before any further work.
